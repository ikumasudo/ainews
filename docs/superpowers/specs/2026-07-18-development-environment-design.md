# 開発環境セットアップ設計

## 目的

アプリケーションの挙動や本番リソースを変更せず、このリポジトリを現在の環境でビルドおよびローカル実行できる状態にする。ツール管理にはプロジェクトローカルの mise 設定と Bun を使用する。

## 対象範囲

- Bun のバージョンを固定するプロジェクトローカルの `mise.toml` を追加する。
- コミット済みの `bun.lock` に従い、解決済みバージョンを変更せず依存パッケージをインストールする。
- 必須の秘密情報 `OPENROUTER_API_KEY` を説明する、安全なサンプルファイルを追加する。
- 秘密値を推測せず、Git 管理対象外のローカル用 `.dev.vars` を空の値で作成する。
- コミット済みのマイグレーションとシードデータを使って、ローカル D1 データベースを初期化する。
- Workers のバインディング型を生成し、ビルドとローカル Worker の起動を検証する。
- 既存文書だけでは不十分な場合、再現可能なセットアップ手順と検証コマンドを文書化する。

本番デプロイ、リモート D1・KV の変更、Cloudflare の認証、OpenRouter API キーの取得は、自動セットアップの対象外とする。これらにはユーザーが所有する認証情報と明示的な権限が必要なためである。

## 実施方式

`mise.toml` をリポジトリ単位のツール定義として使用し、mise 経由で Bun をインストールする。`CLAUDE.md` と `bun.lock` に従い、依存関係の導入と package scripts の実行には Bun を使用する。Wrangler はプロジェクトの開発依存関係として実行するため、Wrangler や TypeScript をグローバルにはインストールしない。

`.dev.vars.example` にはサンプル値のみを記載する。Git 管理対象外の `.dev.vars` は空の値で作成し、秘密情報をリポジトリへ保存しない。これにより通常のローカル HTTP 開発は開始できるが、実際の API キーが設定されるまで、スケジュール実行による AI ニュース抽出は利用できない。

## 検証方法

1. mise が固定された Bun バージョンを有効化することを確認する。
2. ロックファイルを変更せず依存パッケージをインストールする。
3. ローカル D1 のマイグレーションとシードデータを適用する。
4. Workers の型定義を再生成し、設定との整合性を確認する。
5. デプロイを伴わない dry-run ビルドを実行する。
6. 開発サーバーを一時的に起動し、HTTP リクエストを受け付けることを確認する。

外部サービスの認証情報がないことだけを原因とする失敗は、他のセットアップ結果と分けて報告する。架空の認証情報や代替値による回避は行わない。

## 安全性

- 秘密情報を出力またはコミットしない。
- 対話的な外部サービス認証やデプロイを行わない。
- リモートの Cloudflare リソースを変更しない。
- 作業範囲外のワークツリー変更を保持する。
